# 多维度评分方案 - 综合分析总结

## 目的

将当前 binary 评分（0/100）升级为多维度细粒度评分，为 benchmark 提供更有价值的模型能力评估。

## 各维度可行性总结

### 1. Backend-Dev (BigCodeBench)

**可行性**: ⭐⭐⭐⭐ 高

**推荐维度**（按优先级）:

| 维度 | 权重 | 实现难度 | 工具支持 |
|---|---|---|---|
| 功能正确性（细化） | 40% | ⭐⭐ 简单 | unittest, hypothesis |
| 性能效率 | 25% | ⭐⭐⭐⭐ 复杂 | timeit, cProfile, AST |
| 代码风格 | 15% | ⭐⭐ 简单 | pylint, flake8 |
| 鲁棒性 | 10% | ⭐⭐⭐ 中等 | AST, bandit |
| 架构设计 | 5% | ⭐⭐⭐⭐ 复杂 | radon, LLM |
| 安全性 | 3% | ⭐⭐⭐ 中等 | bandit, semgrep |
| 可扩展性 | 2% | ⭐⭐⭐⭐⭐ 极难 | LLM |

**实施路线**:
- **Phase 1（5-7天）**: 功能正确性细化 + 代码风格 + 基础性能检测
- **Phase 2（15-20天）**: LLM 辅助的鲁棒性 + 架构评分
- **Phase 3（20-30天）**: 自动化边界测试 + 语义相似度

**预期收益**: 从 2 级评分（0/100）提升到 50 级细粒度评分

---

### 2. Frontend-Dev (FrontCode)

**可行性**: ⭐⭐⭐⭐⭐ 极高

**推荐维度**（按优先级）:

| 维度 | 权重 | 实现难度 | 工具支持 |
|---|---|---|---|
| 功能完整性 | 30% | ⭐⭐⭐ 中等 | Jest, Playwright |
| HTML 语义化 | 20% | ⭐⭐ 简单 | W3C Validator, axe-core |
| 可访问性 | 15% | ⭐⭐ 简单 | axe-core, WAVE |
| CSS 规范与响应式 | 15% | ⭐⭐ 简单 | Stylelint, AST |
| 代码组织 | 10% | ⭐⭐⭐ 中等 | ESLint, LLM |
| 性能优化 | 5% | ⭐⭐⭐ 中等 | Lighthouse |
| 浏览器兼容性 | 5% | ⭐⭐⭐ 中等 | Browserslist |

**前端特有优势**:
- HTML/CSS 验证工具标准化（W3C）
- 可访问性测试工具完善（axe-core）
- 评分维度清晰明确

**实施路线**:
- **阶段 1（3周）**: HTML/CSS 静态分析
- **阶段 2（5周）**: JavaScript 执行测试
- **阶段 3（4周）**: LLM 辅助评分
- **阶段 4（5周）**: Playwright 浏览器测试
- **阶段 5（4周）**: Lighthouse 性能审计

**预期收益**: 功能缺失检测率从 20% → 90%

---

### 3. Reasoning (MATH)

**可行性**: ⭐⭐⭐⭐⭐ 极高（thinking model 的天然优势）

**推荐维度**（按优先级）:

| 维度 | 权重 | 实现难度 | 依赖 |
|---|---|---|---|
| 答案正确性 | 40% | ⭐⭐ 简单 | 现有 MathScorer |
| 推理完整性 | 25% | ⭐⭐ 简单 | reasoning_content 分析 |
| 推理正确性 | 20% | ⭐⭐⭐ 中等 | LLM-as-a-Judge |
| 方法优雅度 | 10% | ⭐⭐⭐ 中等 | 学科关键词检测 |
| 难度适应性 | 5% | ⭐⭐ 简单 | level 分级标准 |

**核心优势**: `reasoning_content` 是天然解题过程，可直接评分

**实施路线**:
- **Phase 1（立即）**: 答案正确性 + 推理完整性（无需 LLM Judge）
- **Phase 2（1-2周）**: 添加 LLM-as-a-Judge 检查推理正确性
- **Phase 3（持续）**: 方法优雅度 + 难度适应性

**预期效果示例**:
- 完美解答: 95.5 分
- 运气好（蒙对）: 56.5 分  
- 符号错误（思路对）: 62.5 分

**成本**: 单次评测 +$0.015-0.03（主要是 LLM Judge）

---

### 4. System-Architecture (MMLU-Pro)

**可行性**: ⭐⭐⭐ 中等（多选题格式限制）

**推荐维度**（按优先级）:

| 维度 | 权重 | 实现难度 | 备注 |
|---|---|---|---|
| 答案正确性 | 30% | ⭐⭐ 简单 | 现有 ChoiceMatchScorer |
| 推理完整性 | 25% | ⭐⭐ 简单 | 正则匹配，无需 LLM |
| 选项分析深度 | 20% | ⭐⭐ 简单 | 检查是否分析错误选项 |
| 推理置信度 | 15% | ⭐⭐ 简单 | 检测不确定性表述 |
| 学科适应性 | 10% | ⭐⭐ 简单 | 不同学科不同期望长度 |

**核心限制**: 
- 多选题 reasoning 不可信（可能是"合理化"而非真实推理）
- 无法评估"方法优雅度"（没有"方法"可言）
- 某些维度（如推理正确性）不可行

**实施路线**:
- **Phase 1**: 答案正确性 + 推理完整性
- **Phase 2**: 选项分析深度 + 推理置信度
- **Phase 3**: 学科适应性

**关键建议**: 不要对多选题期望过高，专注于可验证的维度

## 总体实施方案

### 第一阶段（2-3周）：立即可行

**目标**: 低投入高回报，快速见效

1. **Backend**: 细化功能正确性评分
2. **Frontend**: HTML/CSS 静态分析
3. **MATH**: 答案正确性 + 推理完整性
4. **MMLU-Pro**: 答案正确性 + 推理完整性

**预期投入**: 10-15 人天
**预期收益**: 
- 区分度从 2 级 → 20-30 级
- 诊断价值显著提升
- 成本增加 < $0.01/次

---

### 第二阶段（1-2月）：中等投入

**目标**: 引入 LLM 辅助评分，覆盖主观维度

1. **Backend**: 鲁棒性 + 架构 LLM 评分
2. **Frontend**: 动态测试 + LLM 评分
3. **MATH**: LLM-as-a-Judge 检查推理正确性
4. **MMLU-Pro**: 选项分析深度 + 推理置信度

**预期投入**: 30-40 人天
**预期收益**:
- 区分度提升到 40-50 级
- 覆盖代码质量主要维度
- 成本增加 $0.02-0.05/次

---

### 第三阶段（2-3月）：长期优化

**目标**: 工业级评分体系

1. **Backend**: 自动化边界测试 + 性能复杂度推断
2. **Frontend**: 浏览器兼容性测试 + Lighthouse 集成
3. **MATH**: 方法优雅度 + 学科定制
4. **MMLU-Pro**: 学科适应性 + 动态权重

**预期投入**: 40-60 人天
**预期收益**:
- 达到人工评审 80% 准确率
- 行业领先的代码评测能力
- 成本增加 $0.1-0.2/次

## 风险与挑战

### 技术风险

1. **LLM 评分一致性**: 同一代码多次评分可能不一致
   - 缓解: 多模型投票、建立评估基准
   
2. **代码执行安全**: 执行 LLM 生成的代码可能带来风险
   - 缓解: Docker 沙箱隔离、资源限制

3. **工具链集成复杂**: 需要集成多个外部工具
   - 缓解: 分阶段实施、优先使用成熟工具

### 成本风险

| 阶段 | 单次成本增加 | 15题成本 | 月度成本（100次评测） |
|---|---|---|---|
| Phase 1 | <$0.01 | $0.15 | $15 |
| Phase 2 | $0.02-0.05 | $0.30-0.75 | $30-75 |
| Phase 3 | $0.1-0.2 | $1.50-3.00 | $150-300 |

### 游戏化风险

- LLM 可能学习"优化评分"而非"写好代码"
- 缓解: 定期更新评分标准、保留人工抽检

### 向后兼容

- 保留原有的 `passed` 字段（0/100 binary）
- 新维度存储在 `details` 字段

## 最终建议

### 推荐实施顺序

1. **Week 1-2**: Frontend HTML/CSS 评分（工具最成熟，见效最快）
2. **Week 3-4**: Backend 功能细化 + MATH Phase 1
3. **Week 5-6**: MMLU-Pro Phase 1
4. **Week 7-10**: Frontend 动态测试 + LLM 评分试点
5. **Week 11-14**: Backend LLM 评分 + MATH Phase 2

### 成功指标

- **区分度**: Top 10% 与 Bottom 10% 分数差距 > 30
- **一致性**: 同一模型多次评测标准差 < 5
- **满意度**: 80% 用户认为新评分更有用
- **效率**: 单次评测时间增加 < 50%

### 关键结论

1. **完全可行**: 多维度评分在技术上完全可行，且有显著收益
2. **差异化管理**: 不同维度需要不同的策略（Frontend 工具最成熟，MATH 天然优势）
3. **渐进式实施**: 从低投入高回报的维度开始，逐步扩展
4. **平衡成本与效果**: Phase 1-2 性价比最高，Phase 3 需要权衡
